1-2 함수 오버로딩(Function Overloading)

함수 호출 시 전달되는 인자를 통해 호출하고자 하는 함수의 구분이 가능하기 때문에 매개변수의 선언 형태가 다르면 동일한 이름의 함수 정의를 허용한다. - C언어는 함수 이름만 이용해 호출 대상을 찾기 때문에 함수 오버로딩이 불가능하다. C++은 함수의 이름, 매개변수의 선언을 이용해 호출 대상을 찾아 오버로딩이 가능하다. 

1-3 매개변수의 디폴트 값(Default Value)

디폴트 값은 함수의 선언 부분에만 표한하면 된다. 

```cpp
int Func(int num1 = 10, int num2 = 20);
```

문제 - 이 형태의 오버로딩이 문제가 있는 이유는?

```cpp
int Func(int a = 10)
{
		return a + 1;
}

int Func(void)
{
		return 10;
}
```

→ 오버로딩의 조건은 만족하지만, 인자를 전달하지 않고 함수를 호출하면 둘 다 호출하기 때문에 컴파일 에러가 발생한다.

# [CA/OS] 

### 오늘 배운 것

![image](https://github.com/user-attachments/assets/bc359d26-2622-4345-9d57-f27af6d731a1)


CPU 스케줄링 - OS가 프로세스들에게 공정하고 합리적으로 CPU 자원을 배분하는 것, 컴퓨터 성능과도 직결됨

프로세스 우선 순위 

프로세스마다 우선 순위가 다름

입출력 집중 프로세스 - 비디오 작업, 디스크 백업 작업 / 대기 상태에 많이 머무름

CPU 집중 프로세스 - 복잡한 수학 연산, 컴파일, 그래픽 처리 작업 / 실행 상태에 많이 머무름

cf) CPU 버스트와 입출력 버스트(burst) - CPU를 이용하는 작업, 입출력 장치를 기다리는 작업, 프로세스는 일반적으로 CPU 버스트와 입출력 버스트를 반복하며 실행된다. 

OS는 프로세스마다 우선순위를 부여, PCB에 우선 순위 명시, PCB에 적힌 우선순위를 기준으로 먼저 처리할 프로세스 결정.

cf) 우선순위가 높은 대표적인 프로세스 - 입출력 작업 많은 프로세스, 실시간 프로세스, 일부 백그라운드 프로세스 등 

스케줄링 큐

OS가 일일이 모든 PCB를 검사하며 먼저 자원을 이용할 프로세스를 결정하는 일은 매우 번거롭고 시간 오래 걸림 

→ 스케줄링 큐로 구현하고 관리 - 프로세스들 줄 세우기 

준비 큐 - CPU를 이용하고 싶은 프로세스들이 서는 줄

대기 큐 - 입출력 장치를 이용하기 위해 대기 상태에 접어든 프로세스들이 서는 줄

OS는 PCB들이 큐에 삽입된 순서대로 프로세스를 하나씩 꺼내어 실행하되, 그 중 우선순위가 높은 프로세스를 먼저 실행함

입출력이 완료되어 인터럽트가 발생하면 OS는 대기 큐에서 작업이 완료된 PCB를 찾고, PCB를 준비 상태로 변경한 뒤 대기 큐에서 제거함. 해당 PCB는 준비큐로 이동


선점형과 비선점형 스케줄링

현재 대부분의 OS는 선점형 스케줄링 사용

선점형 스케줄링

하나의 프로세스가 자원을 사용하고 있을 때 다른 프로세스가 해당 자원을 빼앗을 수 있는 스케줄링

장 - 어느 한 프로세스의 독점을 막고 프로세스들에 골고루 자원을 배분

단 - 문맥 교환 과정에서 오버헤드 발생

비선점형 스케줄링

빼앗을 수, 끼어들 수 없는 스케줄링

장 - 오버헤드가 선점형 스케줄링보다 적음

단 - 모든 프로세스가 골고루 자원을 사용할 수 없음
