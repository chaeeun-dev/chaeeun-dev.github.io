## vector와 list

**std::vector**

size, capacity

삽입과 삭제

- 시작 - push_front O(N)
- 중간 O(N)
- 끝 push_back O(1)

front 첫 번째 원소에 접근 O(1)

 back 마지막 원소에 접근 O(1) 

임의 접근 가능 v[i]



**list**

size, capacity는 없는 개념

삽입과 삭제

- 시작 - push_front O(1)
- 중간 O(1)
- 끝 - push_back O(1)

  → 벡터보다 빠른 것 같아 보이지만 위치를 알고 있을 때 O(1)인 것

front 첫 번째 원소에 접근 O(1)

 back 마지막 원소에 접근 O(1) 

임의 접근 불가능, 원소에 접근하려면 순차 접근 필요




## 이진 탐색, 트리, map, hash_map

**이진 탐색(Binary Search)**

장점 : 검색하는 속도가 빠름 O(logN)

단점 : 데이터가 정렬되어 있어야 함, 중간 삽입과 삭제 할 때마다 정렬을 유지해야 함

→ 이진 탐색 트리



**이진 탐색 트리(Binary Search Tree)**

구조 : 왼쪽 자식 노드는 부모 노드보다 작고, 오른쪽 자식 노드는 부모 노드보다 큼.

탐색 : 찾는 값이 루트 노드보다 크면 오른쪽, 작으면 왼쪽으로 이동하며 반복.

장점 : 중간 삽입과 삭제가 벡터보다 쉬움

단점 :  균형이 깨지면 연결 리스트처럼 변하여 O(N)이 될 수 있음

→ Red Black Tree



**레드 블랙 트리(Red Black Tree)**

특징 : 이진 탐색 트리의 균형이 깨지는(데이터가 한 쪽으로 쏠림) 문제를 해결하기 위해 고안됨.

장점 : 균형을 유지하여 모든 연산이 O(logN)에 수행됨

사용 예 : std::map의 기반이 됨.




**std::map**

구현 방식 : 레드 블랙 트리를 기반

정렬 여부 : 키가 정렬된 상태로 저장

시간 복잡도 :

- 추가 m.insert(make_pair(10, 100)); - O(logN)
- 삭제 m.erase(10); - O(logN)
- 찾기 m.find(10); - O(logN)




**std::unordered_map(hash_map)**

메모리를 내어주고 성능을 얻는 방법

구현 방식 : hash table 기반

정렬 여부 : 키가 정렬되지 않음

시간 복잡도(평균) :

- 추가 - O(1)
- 삭제 - O(1)
- 찾기 - O(1)

최악의 경우 : 모든 키가 같은 해시 값을 가질 때 O(N)이 될 수 있음.

장점 : 메모리를 더 사용하여 빠른 검색 성능 제공

단점 : 데이터가 정렬되지 않음, 메모리를 더 사용함. 

map과 사용법이 거의 동일함. 

→ map - 정렬된 데이터가 필요하거나, 범위 기반 검색이 필요할 때, 

hash_map - 빠른 검색, 삽입, 삭제가 필요할 때 사용 



### 요약

- **이진 탐색**: 정렬된 데이터에 대해 빠르게 검색할 수 있는 알고리즘.
- **이진 탐색 트리**: 삽입과 삭제가 용이한 트리 구조지만, 균형이 깨질 수 있음.
- **레드 블랙 트리**: 이진 탐색 트리의 균형 문제를 해결한 구조.
- **`std::map`**: 레드 블랙 트리를 기반으로 한 정렬된 연관 컨테이너.
- **`std::unordered_map`**: 해시 테이블을 기반으로 한 비정렬 연관 컨테이너.
